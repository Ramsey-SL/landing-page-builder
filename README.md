# Landing Page Builder

Strategy Labs tool that ingests a Shopify collection/product page and rebuilds it
as a **standalone, zero-bloat static page** deployed to Netlify — targeting
Lighthouse **90+** across all four categories while staying a faithful visual
clone of the original (including product hover-image swaps and deferred
conversion tracking).

The original Shopify pages carry ~18 MB of unused JS, ~11 s mobile Total Blocking
Time, and fail accessibility/SEO. We extract the content, rebuild on clean
semantic HTML/CSS with **zero render-blocking JS**, and host on Netlify.

## Pipeline

```bash
node tools/scrape.js <url> <slug>   # 1. Extract products/hero/meta -> output/<slug>/source.json
node tools/build.js  <slug>         # 2. Download+WebP images, inject into template -> index.html
node tools/audit.js  <slug>         # 3. Lighthouse gate (exits 1 if any score < 90)
node tools/deploy.js <slug>         # 4. Deploy to Netlify site sl-<slug> (needs auth, see below)
node tools/serve.js  <slug> [port]  # local preview server
```

Example (full run for a new PNW collection):

```bash
node tools/scrape.js https://thegreatpnw.com/collections/headwear pnw-headwear
node tools/build.js pnw-headwear
node tools/audit.js pnw-headwear            # add --desktop for desktop form factor
node tools/deploy.js pnw-headwear
```

## Architecture

Each page is the merge of two inputs:

1. **`output/<slug>/source.json`** — auto-scraped per page (products, hero, meta,
   nav links, body text). Regenerated by `scrape.js`.
2. **`config/<slug>.json`** — curated "chrome": brand colors/fonts, announcement
   bar, mega-menu nav tree, category strip, footer columns, social links,
   newsletter copy, and tracking IDs. Hand-maintained (brand-level; reuse across
   a brand's collections by copying the file).

Brand assets that can't be regenerated (logo SVG, display font woff2) live in
**`config/brand-assets/<slug>/`** and are committed.

```
tools/
  scrape.js        product/hero/meta extraction (Puppeteer). Card detection walks
                   up to the largest ancestor holding ONE product (theme-agnostic);
                   captures a 2nd card image as imageSrcHover for the hover swap.
  build.js         downloads images -> WebP (sharp), copies brand assets, injects
                   template, validates output. --no-tracking to omit pixels.
  audit.js         serves output/<slug>/ locally, runs Lighthouse via Puppeteer's
                   Chrome, prints score table, saves lighthouse.json, exits 1 on fail.
  deploy.js        Lighthouse-gated Netlify deploy from a clean .publish/ staging
                   dir (only index.html + assets). Creates site sl-<slug> if needed.
  serve.js         minimal static preview server.
  render.js        shared HTML helpers (nav, mega-menu, product grid + hover,
                   footer, social SVGs, deferred tracking loader, token replace).
  extract-design.mjs / inspect.mjs   recon helpers (brand colors, fonts, nav/footer,
                                     tracking detection) — used when onboarding a brand.
templates/landing-page.html          the single token-based template.
config/<slug>.json                   per-page chrome config.
config/brand-assets/<slug>/          logo.svg + display font.
output/<slug>/                       generated (gitignored except source/design/deploy json).
```

### Deferred tracking pattern

Marketing pixels (Meta Pixel, GA4 + Google Ads, Klaviyo) are injected by an inline
loader **only after the first user interaction** (scroll/click/keydown) or a 6 s
post-load fallback. Lighthouse's lab run never interacts, so third-party JS stays
out of the measured load (scores stay high) while real users are still tracked.
The footer/promo signup form pushes a Klaviyo `identify` on submit. Toggle off
with `node tools/build.js <slug> --no-tracking`.

Tracking IDs live in `config/<slug>.json` under `tracking`. For The Great PNW we
reuse their own public IDs (Meta `973636306036006`, GA4 `G-3EJ2E7P52W`, Google Ads
`AW-835951824`, Klaviyo `LKmnDZ`).

### Product hover swap

If a product card on the source page has a second image (front→back/alt view),
`scrape.js` records `imageSrcHover`, `build.js` saves `product-N-hover.webp`, and
the template cross-fades to it on `:hover`/`:focus-visible` (gated behind
`@media (hover: hover)` so touch devices just show the primary image).

## Setup / prerequisites

```bash
npm install        # puppeteer (bundles Chrome), lighthouse, sharp, netlify-cli
```

Node 18+ (developed on v24). See `NOTES.md` / CLAUDE.md for the local-machine
Node/Homebrew quirk.

### Netlify deploy auth

`deploy.js` needs the Netlify CLI authenticated. Either:

- `export NETLIFY_AUTH_TOKEN=<personal access token>` (fastest, unattended), or
- `npx netlify login` once.

Sites are created under the **Fulcrum** team (`ramsey-o7de640`) and named
`sl-<slug>`. Note: `netlify deploy --json` throws a spurious 422 in the current
CLI version, so `deploy.js` parses the CLI's text output for the live URL.

## Continuous checks (GitHub Actions)

- **`smoke.yml`** — runs on every push/PR. Fast, no browser: `node --check` on all
  scripts + `tools/smoke-test.mjs` (render-helper assertions + config JSON
  validation). Catches broken code immediately.
- **`lighthouse.yml`** — scheduled weekly (and manually via *Run workflow*). Runs
  `tools/monitor.js`, which Lighthouse-audits the live URLs found in committed
  `output/*/deploy.json` and fails if any score drops below 90. This is the
  ongoing production-performance guard.

Run them locally too: `npm run smoke` and `npm run monitor`.

## Adding a new client/collection

1. `node tools/scrape.js <collection-url> <slug>`
2. Create `config/<slug>.json` (copy an existing one; update brand chrome) and
   `config/brand-assets/<slug>/` (logo + font). For another collection of an
   existing brand, just copy that brand's config + brand-assets.
3. `node tools/build.js <slug> && node tools/audit.js <slug>`
4. `node tools/deploy.js <slug>`
